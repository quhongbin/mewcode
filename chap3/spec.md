# Agent Loop 自主循环 Spec

## 背景

上一章（chap2）为 MewCode 配齐了六大工具（读文件、写文件、改文件、执行命令、找文件、搜内容），当前 Agent 只能做硬编码的两轮交互——调 LLM、检测到一个工具调用、执行一次、再调一轮 LLM 就结束了。模型无法自主决定下一步动作，每完成一步都需要用户重新发消息催促，无法完成需要多步骤协作的复杂任务。

## 目标

- 将 Agent 从"一步一停"升级为 ReAct 自主循环，能连续思考、行动、观察直到任务完成
- 通过异步事件流彻底解耦 Agent 编排层和界面层
- 支持多工具并行执行以提升效率
- 引入 Plan Mode 让模型先探索规划再执行

## 功能需求

- F1: **Agent 自主循环** — Agent 在收到用户消息后，自动进入 ReAct 循环：调用 LLM 获取响应 → 检测是否包含工具调用 → 若有，执行工具并将结果回灌对话历史 → 再次调用 LLM → 重复，直到模型不再请求工具（返回纯文本回复）。整个过程中 Agent 持续产出事件，外部只需消费事件流。

- F2: **多种停止条件** — 循环在以下任一条件满足时终止：
  - 模型返回纯文本（不含工具调用），视为任务完成
  - 达到迭代上限（可配置的整数，默认 25 轮），作为兜底安全网
  - 用户通过 context 取消（如按 Ctrl+C）
  - 模型调用了注册中心中不存在的工具名
  - LLM 流式调用出错（网络错误、API 错误等）

- F3: **异步事件流** — Agent 运行期间通过一条异步 channel 产出结构化事件，与界面层彻底解耦。事件类型包括：
  - **文本块**（text）：模型输出的文本片段，用于实时流式显示
  - **思考块**（thinking）：模型思考过程的文本片段
  - **工具调用**（tool_call）：模型请求执行某个工具，包含工具名和参数
  - **工具结果**（tool_result）：工具执行完毕后的结果，包含输出内容和是否出错
  - **迭代进度**（iteration）：每轮 LLM 调用开始时发出，包含当前迭代轮次
  - **完成**（done）：循环结束信号，包含结束原因（正常完成/达到上限/被取消/出错）

- F4: **流式双路收集** — 每轮 LLM 流式响应同时做两件事：
  - **实时推送**：将每个文本块/思考块通过事件流推给界面，实现逐字显示
  - **完整累积**：在内部累积出本轮完整的助手文本和工具调用列表，供循环结束后回灌对话历史

- F5: **并行工具执行** — 当模型在一次响应中返回多个工具调用时：
  - 按原始顺序将相邻的只读工具（ReadOnly=true）合并为一组，相邻的有副作用工具各自单独成组
  - 只读组内若工具数超过并行上限（可配置，默认 5），则按上限拆分为子组，子组间串行执行
  - 有副作用组逐个串行执行
  - 组与组之间按原始顺序串行执行
  - 例如并行上限=2，模型返回 `[ReadA, ReadB, ReadC, WriteD]`，执行顺序为：`[ReadA∥ReadB] → [ReadC] → [WriteD]`
  - 每个工具执行完毕后各自产出 tool_result 事件

- F6: **Plan Mode 两段式** —
  - Agent 维护一个模式状态，分为 Plan 模式和 Act 模式
  - **Plan 模式**：仅向模型暴露只读工具（ReadFile、Glob、Grep），模型只能探索代码和制定计划
  - **Act 模式**：向模型暴露全部工具，模型可以执行任何操作
  - Agent 对外提供切换接口，由外部（TUI 层）在用户输入 `/do` 时调用，将模式从 Plan 切换为 Act
  - 默认启动模式为 Act 模式（保持当前行为不变）

## 非功能需求

- N1: **向后兼容** — Agent Loop 改造不应破坏现有的对话流程和工具调用功能。当模型第一轮就返回纯文本时，行为与改造前完全一致。

- N2: **默认安全** — 迭代上限默认为 25 轮，作为兜底安全网，防止模型陷入无限循环。该值可通过 Agent 配置调整。

- N3: **可取消性** — Agent 循环的每一步都检查 context 是否已取消，确保用户 Ctrl+C 能及时终止循环。取消后 Agent 产出 done 事件并优雅退出，不留下残余 goroutine。

- N4: **错误隔离** — 单个工具执行失败不应导致整个循环崩溃。工具错误以 tool_result（isError=true）的形式回灌对话历史，让模型自行决定如何处理。

## 不做的事

- 不做权限系统（哪些工具需要用户确认才能执行）
- 不做上下文压缩（对话历史过长时的裁剪/摘要）
- 不做用户交互式确认（工具执行前的二次确认弹窗）
- 不做多用户消息排队（用户在前一轮未完成时发送新消息的处理）
- 不做 Token 用量统计（Provider 层当前未返回 usage 信息）

## 验收标准

- AC1: 发送一条需要多步操作的消息（如"读取 a.go 然后创建 b.go"），Agent 自动循环多轮直到任务完成，无需用户再次发消息。（对应 F1）

- AC2: 当模型连续请求工具达到迭代上限时，循环停止并产出 done 事件，结束原因为达到上限。（对应 F2）

- AC3: 在 Agent 循环运行中取消 context，循环在当前步骤完成后停止，不启动下一轮 LLM 调用。（对应 F2、N3）

- AC4: 模型调用一个不存在的工具名时，循环停止并产出 done 事件，结束原因为未知工具。（对应 F2）

- AC5: TUI 层仅通过消费事件流来渲染界面，不直接调用 Agent 的内部方法。（对应 F3）

- AC6: 并行上限设为 2 时，模型一次返回 3 个只读工具调用，前 2 个并发执行，第 3 个等前 2 个完成后再串行执行；返回 2 个写工具调用时，按顺序串行执行。（对应 F5）

- AC7: Plan 模式下模型只能看到只读工具；切换为 Act 模式后模型可以看到全部工具。（对应 F6）

- AC8: 模型返回纯文本（无工具调用）时，行为与改造前一致，界面正常显示完整回复。（对应 N1）
