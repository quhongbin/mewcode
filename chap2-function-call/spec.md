# 工具系统 Spec

## 背景

MewCode 当前是一个纯聊天机器人：用户发消息，模型流式回复文本，对话历史持续累积。模型无法与用户的实际工作环境交互——不能读代码、不能写文件、不能执行命令。

本次要为 MewCode 装上工具系统，使模型能够识别需要工具辅助的场景，生成结构化的工具调用请求，由宿主程序执行后将结果回灌，模型据此生成最终回复。这是从「聊天机器人」到「能干活的 Agent」的关键一步。

## 目标

- **模型具备操作能力**：模型可以通过工具读写文件、执行命令、搜索代码，不再只能输出文本
- **统一且可扩展的工具体系**：所有工具遵循同一接口，新增工具只需实现接口并注册，无需修改其他代码
- **为 Agent Loop 预留架构**：当前实现单轮单工具调用，但架构分层设计使得后续引入 ReAct 循环时只需在编排层加循环，不动底层

## 功能需求

- **F1: 统一工具接口** — 每个工具通过统一的接口描述自己：包含名称、人类可读的描述、参数定义（JSON Schema 格式）、执行入口，以及工具元数据。工具元数据包含三个属性：
  - **分类**（category）：标识工具所属领域，如 filesystem、shell、search
  - **只读标记**（read_only）：标识该工具是否只读操作
  - **破坏性标记**（destructive）：标识该工具是否可能造成不可逆变更
  
  元数据供编排层和 UI 使用，所有工具包括后续新增的都必须提供。

- **F2: 六大核心工具** — 提供以下六个开箱即用的工具：
  - **读文件**：给定文件路径，返回带行号前缀的文件内容（如 `  42│func main() {`），以便模型后续直接引用代码位置。支持 offset（从第几行开始）和 limit（读几行）参数，让模型分段读取大文件。二进制检测：先读取前 512 字节，若包含 NUL 字符（`\x00`）则判定为二进制文件，拒绝读取并返回提示让模型改用命令行工具处理。
  - **写文件**：给定文件路径和内容，创建或覆盖文件。若父目录不存在则递归创建（目录权限 0755），文件权限 0644。成功后返回结构化消息：`[成功写入 N 字节到 path]`。
  - **编辑文件**：给定文件路径、原文片段和替换内容，在文件中唯一匹配原文片段后替换。匹配不到返回"未找到匹配"错误；匹配到多处返回"找到 N 处匹配，请提供更多上下文"错误。
  - **执行命令**：给定命令字符串，在宿主 shell 中执行并返回标准输出和标准错误。
  - **查找文件**：给定 glob 模式，返回匹配的文件路径列表。
  - **搜索代码**：给定正则表达式和可选的搜索路径，返回匹配的行及其上下文。

- **F3: 工具注册中心** — 提供一个集中登记所有工具的注册中心。可按名称查找工具，可将已注册工具列表转换为 LLM API 所需的工具定义格式。

- **F4: 工具执行安全** — 每个工具执行受超时控制。执行失败（超时、运行时错误、参数校验失败）时，返回结构化的错误信息给模型，而非抛出异常或崩溃。模型可据此调整策略重试。

- **F5: 流式工具调用解析** — LLM 客户端能从流式响应中识别工具调用事件，将分片到达的 JSON 参数拼接为完整参数。工具调用作为流式数据块的一种新类型传递给上层。

- **F6: 对话中的工具编排** — 用户发消息后，若模型的响应包含工具调用，系统自动执行该工具，将结果作为工具结果消息回灌到对话历史，再请求模型生成最终文本回复。整个过程对用户透明——用户在界面上看到工具执行的过程和最终回复。

- **F7: 工具调用过程可视化** — 终端界面展示工具调用过程：显示正在调用的工具名称和参数摘要，执行完成后显示结果摘要，让用户了解模型在做什么。

## 非功能需求

- **N1: 双供应商兼容** — 工具系统同时支持 Anthropic 和 OpenAI 两种协议的工具调用格式，上层编排逻辑不感知底层协议差异。
- **N2: 可扩展性** — 新增一个工具只需：实现工具接口 + 在注册中心注册。不需要修改 Provider、Conversation、TUI 或编排层的任何代码。
- **N3: 容错性** — 工具执行的任何失败都转化为结构化结果返回给模型，不会导致程序崩溃或对话中断。
- **N4: 流式体验** — 工具调用的 JSON 参数分片到达时实时拼接，不阻塞 UI。工具执行阶段 TUI 保持响应，用户可通过 Ctrl+C 取消。
- **N5: Token 效率** — 工具返回的结果保持简洁结构化，避免将大量无关内容灌入对话历史。

## 不做的事

- 不做 Agent Loop（ReAct 循环）— 单轮单工具，自动多轮循环留到下一章
- 不做工具调用权限控制 — 无确认弹窗、命令白名单或审批机制
- 不做沙箱隔离 — 工具直接在用户真实环境中运行
- 不做并发工具执行 — 不涉及多工具并行调度
- 不做工具结果的流式回传 — 结果完整返回后一次性回灌

## 验收标准

- **AC1（F1）：** 任意工具实例可被统一接口调用，传入 JSON 参数后返回结构化结果；工具的元数据（分类、只读、破坏性）可通过接口读取。
- **AC2（F2-读文件）：** 读取文本文件返回带行号内容；offset=10 limit=5 只返回第 10-14 行；读取二进制文件返回拒绝提示。
- **AC3（F2-写文件）：** 写入不存在的深层路径文件，父目录自动创建，返回成功消息。
- **AC4（F2-编辑文件）：** 唯一匹配替换成功；不存在的原文返回"未找到匹配"；重复内容返回"找到 N 处匹配"。
- **AC5（F2-执行命令）：** `echo hello` 返回 `hello`；不存在的命令返回结构化错误。
- **AC6（F2-查找文件）：** `*.go` 模式返回所有 Go 文件路径。
- **AC7（F2-搜索代码）：** 正则搜索返回匹配行及上下文。
- **AC8（F3）：** 注册后按名查找成功；获取工具列表返回所有工具的 API 格式定义。
- **AC9（F4）：** 超时工具返回超时错误而非卡死；错误信息可理解。
- **AC10（F5）：** 分片到达的 JSON 参数碎片最终拼接为完整工具调用参数。
- **AC11（F6）：** 发送需要工具的请求，模型调用工具，系统执行并回灌结果，模型基于结果给出文本回复，一次交互完成。
- **AC12（F7）：** 工具执行期间终端显示工具名称和参数摘要；完成后显示结果摘要。
- **AC13（集成）：** 编译无错误，测试全部通过；不使用工具时行为与改造前完全一致。
